Highest Priority

Daily Pull Request Workflows

These cards focus on the full PR tasks developers actually do: branch from the latest code, commit and push, open the request, handle feedback, and finish the merge cleanly.

Start Work

Create a Feature Branch

Branch 
git checkout main
git pull
git checkout -b feature/login
git branch

Start from an updated main branch so your feature work begins from the latest shared code before you open a PR.

Save Work

Commit and Push Your Changes

Git 
git status
git add .
git commit -m "Add login form"
git push -u origin feature/login

This is the normal sequence for publishing your work to GitHub so the branch is ready for a Pull Request.

Submit

Open a Pull Request

Review 
git push -u origin feature/login

GitHub:
1. Click "Compare & pull request"
2. Set base branch to main
3. Set compare branch to feature/login
4. Write the title and description
5. Click "Create Pull Request"

The branch must already exist on GitHub. After that, verify the base and compare branches before you create the Pull Request.

Review Cycle

Respond to Review Feedback

Feedback 
git status
git add .
git commit -m "Address review comments"
git push

You update the same Pull Request by committing additional fixes to the same branch and pushing them again.

Finish Work

Merge and Clean Up

Merge 
Merge Pull Request on GitHub
git checkout main
git pull
git branch -d feature/login
git push origin --delete feature/login

After the Pull Request is merged, update your local main branch, remove the local feature branch, and delete the remote branch if your team does that manually.

Collaboration

What Is a Pull Request?

A Pull Request, commonly called a PR, is a request to merge the changes from one branch into another branch.

Compare Branch feature/login
↓ Pull Request ↓
Base Branch main
01

Review Code

Other developers can inspect the changes before they enter main.

02

Leave Comments

Reviewers can ask questions or suggest improvements.

03

Request Changes

A reviewer can require corrections before approval.

04

Approve Work

A reviewer can confirm that the changes are ready.

05

Merge Safely

Approved work can be combined with the base branch.

Start New Work

Creating a Branch

Create and Switch

git checkout -b feature/login

Creates a branch named feature/login and immediately switches to it.

See Your Branches

git branch

The branch with an asterisk is your current branch.

Switch Branches

git checkout main

Switches from your current branch to main.

Modern Git Alternative

git switch -c feature/login
git switch main

git switch can also create and change branches.

Create your feature branch from an updated copy of main whenever possible. 
git checkout main
git pull
git checkout -b feature/login

Commit Your Work

Making Changes

1

Check Status

git status

See which files were added, changed, or deleted.

2

Stage Files

git add .

Add all changed files to the next commit.

3

Create Commit

git commit -m "Add login form"

Save the staged changes with a meaningful message.

4

Push Branch

git push -u origin feature/login

Publish the branch and connect it to the remote branch.

The -u option sets the upstream branch. After the first push, future updates can normally use only git push. 
git status
git add .
git commit -m "Add login form"
git push -u origin feature/login

Opening a Pull Request

1

Push Your Branch

The branch must exist on GitHub before you can open a Pull Request.

2

Click Compare & Pull Request

GitHub often displays this button after a new branch is pushed.

3

Verify the Branches

Base main
←
Compare feature/login
4

Write the Title and Description

Explain what changed, why it changed, and how the reviewer can test it.

5

Create the Pull Request

Click Create Pull Request to submit it for review.

Simple PR Description

## What changed

* Added the login form
* Added form validation
* Added error messages

## How to test

1. Start the application
2. Open the login page
3. Submit valid and invalid information

## Checklist

* [x] Code works locally
* [x] Tests pass
* [x] I reviewed my own changes

Code Review

Reviewing a Pull Request

Functionality

  • Does the code work?
  • Does it solve the requested problem?
  • Do the tests pass?

Readability

  • Are variable names clear?
  • Are functions focused?
  • Is the code easy to follow?

Code Quality

  • Is there duplicate code?
  • Are unnecessary files included?
  • Are errors handled correctly?

Documentation

  • Is the PR description clear?
  • Are complex decisions explained?
  • Were important instructions updated?

Possible Review Actions

Comment

Leave general feedback without approving or blocking the PR.

Approve

Confirm that the code is ready to merge.

Request Changes

Require specific corrections before the PR is merged.

Respond to Feedback

Updating a Pull Request

You do not need to create a new Pull Request after receiving review feedback. Add more commits to the same branch and push them.

Update Commands

git status
git add .
git commit -m "Address review comments"
git push

What Happens

Change code ↓ Create commit ↓ Push branch ↓ PR updates automatically
Every commit pushed to the compare branch automatically appears in the existing Pull Request.

Complete the PR

Merging

After the Pull Request is approved and required checks pass, the changes can be merged into the base branch.

Full History

Create a Merge Commit

Keeps every feature-branch commit and creates an additional merge commit.

One Commit

Squash and Merge

Combines all Pull Request commits into one commit on the base branch.

Linear History

Rebase and Merge

Adds the feature commits onto the base branch without a merge commit.

Use the merge option required by your team or repository. Do not pick a different method only because it looks cleaner.

After Merging

git checkout main
git pull
git branch -d feature/login

Fix Merge Conflicts

Resolving Conflicts

!
This branch has conflicts that must be resolved.

Git cannot automatically decide which version of the changed code should be kept.

1

Switch to Your Branch

git checkout feature/login
2

Merge Main into It

git merge main
3

Open Conflicted Files

Choose the correct code and remove the conflict markers.

4

Stage the Resolved Files

git add .
5

Complete the Merge

git commit
6

Update the Pull Request

git push

Conflict Markers

<<<<<<< HEAD
# Code from your current branch
=======
Code from the branch being merged
>>>>>>> main
Do not commit the <<<<<<<, =======, or >>>>>>> markers. Resolve the code and remove all markers first.

Git Reference

Useful Commands

Branches

git branch Show local branches
git checkout branch-name Switch branches
git checkout -b new-branch Create and switch
git branch -d branch-name Delete a merged local branch

Remote Changes

git push Upload local commits
git pull Download and merge remote changes
git push -u origin branch Publish a new branch
git push origin --delete branch Delete a remote branch

Changes and Commits

git status See repository status
git add . Stage all changes
git commit -m "message" Create a commit
git log --oneline View concise commit history

Merging

git merge feature-branch Merge a branch into the current branch
git merge --abort Cancel an unfinished merge
git diff See unstaged differences
git diff --staged See staged differences

Vocabulary

Common Terms

Branch

A separate line of development.

PR

An abbreviation for Pull Request.

Review

Feedback given before changes are merged.

Approve

A reviewer accepts the proposed changes.

Merge

Combine changes from one branch into another.

Conflict

Git cannot automatically combine competing changes.

Base Branch

The branch receiving the changes, often main.

Compare Branch

The branch containing the proposed changes.
