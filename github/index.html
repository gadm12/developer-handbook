GitHub Actions Cheat Sheet

A workflow-first reference for creating workflow files, running tests on push and Pull Requests, wiring secrets safely, and diagnosing failed Actions runs without getting lost in the logs.

GitHub Actions Docs →

Highest Priority

Daily GitHub Actions Workflows

These cards group the real CI tasks developers usually do together: create the workflow file, run tests after pushes and Pull Requests, inject secrets safely, and read failures quickly.

Setup

Create a Workflow

YAML 
mkdir -p .github/workflows
touch .github/workflows/tests.yml

name: Test Project
on:
  workflow_dispatch:
jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

Start by creating the required folder path, add a workflow file, then define the minimum structure so GitHub recognizes it and you can run it manually while building it out.

Continuous Integration

Run Tests on Push

Push 
on:
  push:

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 22
      - run: npm install
      - run: npm test

This is the normal push-based CI sequence: start on every push, check out the code, prepare the runtime, install dependencies, and run the test command automatically.

Review Gates

Run Tests for Pull Requests

PR Checks 
on:
  pull_request:
    branches:
      - main

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with:
          python-version: "3.13"
      - run: pip install -r requirements.txt
      - run: pytest

Use this when you want GitHub to validate a branch before it merges into main. The workflow runs whenever the Pull Request is opened or updated.

Configuration

Use Secrets and Environment Variables

Secrets 
env:
  NODE_ENV: test

steps:
  - name: Use API key
    run: npm run test:api
    env:
      API_KEY: ${{ secrets.API_KEY }}

Keep sensitive values in GitHub Secrets, define non-sensitive shared values with env, and pass secrets only to the steps that actually need them.

Debugging

Troubleshoot a Failed Workflow

Logs 
GitHub:
1. Open the Actions tab
2. Select the failed workflow run
3. Open the failed job
4. Expand the failed step
5. Read the log output
6. Fix the workflow or project code
7. Push again or re-run the job

Start with the failed step, not the whole file. Most workflow problems are a missing dependency, wrong file path, bad secret, or command that also fails locally.

Automation

What Is GitHub Actions?

GitHub Actions automatically performs tasks whenever a configured event happens in your repository.

✓

Run Tests

Automatically run pytest, npm tests, or another test suite.

F

Check Formatting

Verify that code follows the project's formatting rules.

L

Run Linters

Check code for errors, warnings, and style problems.

B

Build Projects

Compile or prepare an application for production.

D

Deploy Applications

Publish an application after tests and builds succeed.

GitHub Actions is commonly used for continuous integration, often shortened to CI.

File Location

Workflow Location

📁 .github
📁 workflows
YML tests.yml

Required Path

.github/
└── workflows/
    └── tests.yml

GitHub only recognizes workflow files placed inside .github/workflows/.

Placing tests.yml in the project root will not create a GitHub Actions workflow.

Main Sections

Workflow Structure

Basic YAML

name: Test Project

on:
  push:

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: pytest

How It Is Organized

name

The workflow name displayed in the Actions tab.

on

The repository event that starts the workflow.

jobs

One or more groups of related tasks.

steps

The actions and shell commands performed by a job.

YAML uses indentation to define structure. Use spaces consistently and avoid tabs.

Workflow Events

Triggers

Run on Push

on:
  push:

Starts the workflow whenever commits are pushed.

Run on Pull Request

on:
  pull_request:

Starts the workflow when a pull request is opened or updated.

Run on Both

on:
  push:
  pull_request:

Runs for pushed commits and pull requests.

Run Manually

on:
  workflow_dispatch:

Adds a button in the Actions tab for manually starting the workflow.

Only Run on Main

on:
  push:
    branches:
      - main

Run on Multiple Branches

on:
  push:
    branches:
      - main
      - development

Groups of Work

Jobs

Job Example

jobs:
  test:
    runs-on: ubuntu-latest

What It Means

jobs:

Starts the jobs section.

test:

The custom name of this job.

runs-on:

Selects the machine that runs the job.

ubuntu-latest

Uses a GitHub-hosted Ubuntu runner.

Multiple Jobs

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - run: pytest

  lint:
    runs-on: ubuntu-latest
    steps:
      - run: pylint src
Jobs normally run independently. A workflow can test, lint, build, and deploy using separate jobs.

Individual Tasks

Steps

Use an Existing Action

steps:
  - uses: actions/checkout@v4

The uses keyword runs a reusable action.

Run a Shell Command

steps:
  - run: pytest

The run keyword executes a command on the runner.

Name Your Steps

steps:
  - name: Download repository
    uses: actions/checkout@v4

  - name: Run tests
    run: pytest

Descriptive step names make workflow logs easier to understand.

Reusable Actions

Common Actions

Repository

Checkout Code

- uses: actions/checkout@v4

Downloads your repository code onto the workflow runner.

Python

Setup Python

- uses: actions/setup-python@v5
  with:
    python-version: "3.13"

Installs the selected Python version.

Node.js

Setup Node

- uses: actions/setup-node@v4
  with:
    node-version: 22

Installs the selected Node.js version.

The checkout step normally comes first because later steps need access to your repository files.

Pytest

Python Example

name: Python Tests

on:
  push:

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - name: Checkout repository
        uses: actions/checkout@v4

      - name: Setup Python
        uses: actions/setup-python@v5
        with:
          python-version: "3.13"

      - name: Install dependencies
        run: pip install -r requirements.txt

      - name: Run pytest
        run: pytest
1

Checkout

Download the repository code.

2

Setup Python

Install the requested Python version.

3

Install

Install packages from requirements.txt.

4

Test

Run the pytest test suite.

The workflow fails if requirements.txt does not exist or if pytest is not installed.

Test

Workflow files use YAML and normally end in .yml or .yaml.

Workflow files live inside .github/workflows/.

03

A workflow commonly contains name, on, jobs, and steps.

04

05

The run keyword executes a shell command.

06

07

Failed workflow logs show which job and step caused the problem.
